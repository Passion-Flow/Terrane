## 1. 问题陈述（Problem Statement）

**痛点**：2026 年 4 月 karpathy 发布 llm-wiki 方法论（"用 LLM 把知识增量编译成持久 wiki，而非每次查询临时 RAG"），推文原话"我最近的大部分 token 吞吐不再用于操作代码，而是用于操作知识"——引爆全网：数月内 20+ 开源复刻（最大 58k star），中文社区现象级讨论。但：

- **现成方案全是手工拼装**：Obsidian + Claude Code + git 的组合非技术用户根本玩不动；复刻项目全是单机 demo/CLI/插件，无一做成企业级私有化商业产品（已核验：截至 2026-06-13 无任何此类成熟商业产品挂牌，但商业化雏形已现——Synthadoc"Community Edition"+AGPL 是教科书式 open-core 铺垫，**窗口期按月计不按季度计**）。
- **传统 RAG 产品不做知识沉淀**：AnythingLLM/RAGFlow/FastGPT/MaxKB 每次查询从零检索，没有"一次编译、持久沉淀、知识复利"；跨会话记忆全梯队缺位；RAGFlow 有 GraphRAG 却无图谱浏览编辑 UX。
- **开源供给塌方 + 免费基线抬升并存**：Khoj 近 3 月停更、Reor 归档、Quivr 转型、QAnything 停更；同时 Open Notebook（28k star）/腾讯 WeKnora 等开源自托管方案抬高免费基线——商业产品必须靠企业级能力（权限/审计/集群/信创/License 管控）拉开差距，这恰是开源实现普遍缺失的。
- **私有化交付全行业反人类**：企业版获取全靠联系销售/填问卷；Open WebUI 30 天滚动窗口 >50 用户即禁白标（许可页原文已核验）、MaxKB GPL 传染、Smart Connections 涨至 $299/年引发反弹——竞品在系统性制造"被许可证背刺"的迁移人群。
- **数据安全焦虑真实存在**：NotebookLM 体验最好但纯云零私有化；腾讯 ima 绑生态零私有化（其同源引擎 WeKnora 已开源，但为无商业支持的引擎件）。律师/医生/研究者/涉密企业的资料进不了云。

**谁有痛点**：

- **P1 个人专业用户**（律师/医生/研究者/作者/开发者）：资料敏感、量大、终身积累，要"装在自己机器上的 NotebookLM + 第二大脑"，怕云服务关停/涨价/偷数据。
- **P2 企业**（知识密集型团队、央国企信创场景）：要内网部署、人手一个私有知识空间 + 组织级共享、统一管控审计、License 合规采购。
- **P3 运营者**（厂商自己 + 私有化部署方管理员）：要 seat 管理、配额、审计、模型渠道管理便捷。

**不解决会怎样**：用户继续在"云端好用但不敢放数据"与"本地安全但难用"之间二选一；2026-2027 信创替换窗口期错过。

**为什么是我们**：① 七大确证差异化空白（单机集群同镜像 / MCP 双向 / llm-wiki 企业级承接 / 跨会话记忆 / 许可干净可白标 / 页面化 License 激活 / 中文个人数据源覆盖）全部有调研实锤、复核确认 60 天内未被填掉，且与我方既有能力对口；② Forge License 体系 + OpenRelay 网关架构 + Navtra 集群 License 方案直接复用打法（代码按项目独立铁律各自拷贝）；③ 时机：MCP 无状态化方向已定（RC）、Anthropic 定时 Agent 公测——基础设施刚就位，承接者缺位。

## 3. 用户画像（Personas）

- **P1 个人专业用户**：资料 = 文献/案卷/病例/书稿/代码仓/网页剪藏/微信读书笔记。最在意：数据永远在自己手里、中文资料解析准、越用越懂自己、能逃逸（Markdown/git 落盘）。硬件从无 GPU 的 NAS/迷你主机到工作站。
- **P2-a 企业知识工作者（seat 使用者）**：个人空间 + 团队共享库；在 IDE/Claude Code 里也要能查到自己的库（MCP）。
- **P2-b 企业技术决策者**：内网部署、统一管控、审计、信创合规、License 采购走预算流程（买断+维保偏好）。
- **P3 运营者/管理员**：管 seat、配额、模型渠道、连接器凭据、审计日志；要页面化零配置。
- **P4 厂商超管**：License 签发（Forge）、客户健康、续费维保。

## 4. 用户故事 + 验收标准（Given/When/Then）

按模块组织。每条 Story 给 Given/When/Then + 边界。

### 4.1 知识引擎核心（三层架构 + 三流程，llm-wiki 蓝本产品化）

**Story 4.1.1 三层数据模型**
- Given 用户创建一个知识库（Workspace 内可建多个库）
- When 系统初始化该库
- Then 库内存在三层：**Raw 层**（原始源：文件/网页/对话归档，不可变，LLM 只读）、**Wiki 层**（LLM 生成并全权维护的知识页：摘要页/实体页/概念页/对比页/综述页，含 `[[双向链接]]`）、**Schema 层**（库宪法：结构约定/抽取实体类型白名单/页面模板/工作流配置，版本化，用户与 Agent 共同演进）
- 边界：Raw 层任何修改走"新版本追加"不覆盖；Wiki 页默认只读，用户手改走显式"接管编辑"动作并记 append-only 日志；Schema 变更记版本可回滚；`[[链接]]` 限库内（跨库图谱联邦 → v2 预留）
- 依据：[karpathy llm-wiki gist](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f)（已直抓原文核验）

**Story 4.1.2 Ingest 摄入即编译**
- Given 用户向库内投放任意源（上传/剪藏/连接器同步/对话沉淀/Agent 抓取）
- When 摄入管线处理该源
- Then 自动完成：解析 → 生成摘要页 → 创建/更新相关实体页与概念页（验收线：单源摄入必须完成摘要页 + ≥1 个实体/概念页创建或更新）→ 更新库索引 → 追加结构化日志（append-only，`[时间] ingest | 源标题` 格式）→ 增量更新图谱（仅受影响子图）
- 边界：批量摄入提供"弱监督模式"与"逐一确认模式"；失败源停留在 Raw 层标错误原因可重试；摄入前显示 token 估价并受预算闸门约束（4.12.4）
- 依据：llm-wiki Ingest 流程原文；Graphify 零配置增量更新（[量子位](https://www.qbitai.com/2026/04/396983.html)）

**Story 4.1.3 Query 问答回填（知识复利闭环）**
- Given 用户在聊天中获得了一个高质量回答
- When 用户点击"沉淀为知识页"（或 Agent 判断答案含库内没有的新综合结论时主动建议）
- Then 回答被编译为新知识页：自动定标题/选分类/生成双向链接/标注来源/更新索引；回答形态支持 Markdown 页、对比表、图表
- 边界：重复沉淀做内容去重（合并而非新建）；沉淀可撤销；"自动沉淀"默认关；**wiki 页内无源依据的段落必须标注"推断"**（防幻觉固化，R11）
- 依据：llm-wiki"好答案要回填进 wiki"规则——与所有 RAG 竞品的本质区别

**Story 4.1.4 Lint 库体检（差异化独有功能）**
- Given 库已有内容且配置了体检计划（默认每周，可改频率/手动触发）
- When 体检 Agent 运行
- Then 产出"库体检报告"：页面间矛盾陈述、被新源推翻的过时主张、孤立页、缺专属页的概念、缺失交叉引用、可补的数据空白、建议的新调查方向；每项可一键"让 Agent 修复"
- 边界：修复全部以 diff 预览、逐项接受/拒绝、可回滚；体检消耗 token 报告头部披露并受预算闸门约束；检出质量走自建 Lint 评测集（§2）
- 依据：llm-wiki Lint 流程；竞品调研确认市面零产品有此功能

**Story 4.1.5 git 化存储与可逃逸性**
- Given 用户想确认"数据永远是我的"
- When 用户开启库的"本地仓库镜像"（默认开）
- Then Wiki 层全部以 Markdown 文件（含 YAML frontmatter）持续落盘到挂载卷的 git 仓库，每次 Agent 写库 = 一次 commit（可审计可回滚）；该目录可被 Obsidian 直接打开；提供一键全量导出（Markdown + 附件 + 图谱 JSON）
- 边界：外部直接改文件触发"外部变更检测"提示导入合并；**硬删除一致性**——页/库删除时同步删除镜像文件并提交，页删除二次确认（git 历史可恢复，DSR 除外），库删除需输入库名确认 + 提示一键导出；DSR/数据销毁场景对 git 历史执行 history-rewrite 或整仓销毁（对齐 data-destruction.md），已删内容在导出包中不可复现
- 依据：llm-wiki"wiki 即 git 仓库"；可逃逸性是对竞品许可证背刺的反向营销卖点

**Story 4.1.6 库的归属与共享（企业核心场景）**
- Given Workspace 内多名成员
- When 成员创建/共享库
- Then 每个库有 owner；库级可见性三档：**private**（个人空间，默认）/ **shared**（显式邀请成员，库内角色 viewer/editor）/ **workspace**（全租户可读）；共享邀请走站内通知 + 邮件；库级角色与 Workspace 五角色独立授予，**实际权限取两者交集**（全矩阵见 02 设计文档）
- 边界：private 库的检索/图谱/记忆对他人不可见，默认连 Workspace Admin 不可读内容（策略项位于后台 Settings，默认关闭 Admin 可读，改配仅对未来读取生效并全程记审计）；**"不可读"为应用层访问控制承诺，不构成对基础设施操作者的密码学保证——产品文档如实声明**：git 镜像/备份介质含 private 库内容，企业部署指引要求卷与备份加密（密钥由部署方持有），审计日志只记操作元数据不记页面内容，Lint 全局视图对 private 库仅展示统计计数不展示报告正文；shared 库的对话沉淀记忆仍归个人（4.6.2）；库转移 owner 需双方确认并记审计
- 依据：一致性审查 E-1 阻塞项修复 + 再审④权限旁路封口

### 4.2 摄入管线（解析/分块/索引）

**Story 4.2.1 全格式解析**
- Given 用户上传 PDF（含扫描/复杂版式/表格/公式）/ Word / PPT / Excel / Markdown / HTML / EPUB / 图片 / 音频 / 视频 / 代码仓 zip
- When 解析管线处理
- Then 统一产出结构化 Markdown + 资产文件：**中文/复杂版式 PDF 主路锁 MinerU**（pipeline=CPU 档 / VLM 后端=GPU 档，OmniDocBench 95+）；**Docling 仅限 Office/HTML/EPUB 长尾**（其中文 PDF 乱码有官方 issue 实锤——出厂预置中文 OCR 模型与显式 lang 配置，不走中文 PDF 主路）；图片走 PP-OCRv5 + VLM 描述；音视频走 FunASR SenseVoice（中文）/ faster-whisper turbo（多语兜底）转写；代码仓走 tree-sitter AST 本地解析（零 LLM token）
- 边界：解析器按文件类型自动路由、失败自动降级（VLM→pipeline→纯 OCR）并标注降级原因；附件统一转存对象存储、图片本地化下载；**MinerU 版本锁 ≥3.1.0、仅打包 MinerU2.5-Pro 系权重**（旧 2509 权重 AGPL 严禁入包）；解析任务与 LLM 推理在 CPU 档强制互斥排队、MinerU 进程按任务拉起用完即杀（内存账约束，见 4.10.3）
- 依据：调研④+二轮硬件核验——MinerU pipeline 官方内存下限 16GB（README 硬件表原文）；Docling 中文 issue #748/#225/#2866 实锤

**Story 4.2.2 分块与上下文增强**
- Given 解析产物进入索引
- When 分块器运行
- Then markdown-aware 结构分块（512 token + 15% 重叠默认，Schema 可调）；**GPU 或外接模型可用时**可开启 contextual retrieval（LLM 为每块生成 50-100 token 上下文前缀；检测到 GPU/外接模型时默认开，纯 CPU 本地档默认关）——硬件档位差异而非 SKU/拓扑门控
- 边界：表格整体成块不截断；代码块按函数/类边界
- 依据：Anthropic contextual retrieval 实测（检索失败率 -49%、叠重排 -67%）

**Story 4.2.3 检索引擎随库规模自动升档**
- Given 库规模增长
- When chunk 数达到阈值（默认 5 万，库设置可改）
- Then 小库档走"结构化索引 + 全文检索"（零嵌入开销）；达阈值自动启用向量档（pgvector HNSW + halfvec）并后台补嵌入；切换透明、可强制指定
- 依据：llm-wiki"~100 源内靠 index 就够"

### 4.3 检索引擎

**Story 4.3.1 混合检索**
- Given 用户提问或 Agent 需要检索
- When 检索执行
- Then 多路召回融合：召回路数随档位自适应——小库档 = 词法 + AGE 图 1-2 跳邻域 + 最近日志时间线三路；向量档 = 加 pgvector 向量（Qwen3-Embedding）四路全开；RRF（k=60）融合后重排 top-20（CPU 档 BGE-reranker-v2-m3 经 llama.cpp `/v1/rerank` 端点——官方文档点名支持该模型；GPU 档 Qwen3-Reranker-4B）
- 边界：**中文词法召回主路 = zhparser**（2026-06 仍活跃、有 PG17+ 现成包）；pg_trgm 仅作拼写容错/模糊辅路；pg_jieba 不引入（2022 年后无 commit，已核验）；精确串命中词法路保底；权重 Schema 可调；检索范围支持多库勾选（按权限）
- 依据：行业共识 + 二轮核验（zhparser/pg_jieba/llama.cpp rerank 端点全部一手确认）

**Story 4.3.2 带引用回答**
- Given 任何检索增强回答
- When 答案生成
- Then 每个事实陈述带引用角标，点击直达源 chunk 高亮位置（PDF 页内定位）；引用列表区分 Raw 源与 Wiki 页
- 边界：推断显式标注；引用准确率走 §2 抽检协议

### 4.4 知识图谱（双引擎：三元组图 + wiki 物化投影——本产品核心创新）

**Story 4.4.1 LightRAG 增量图构建**
- Given 源完成摄入
- When 图构建器运行
- Then 按 Schema 实体类型白名单抽取带类型实体/关系入 AGE 图（与 pgvector 同库同事务）；代码走 AST、结构化文档走规则路径零 LLM 成本；增量 insert/delete 只重算受影响子图，绝不重建全图
- 边界：抽取用低价模型（网关角色路由），gleaning ≤ 1 轮；1MB 文档抽取预算口径 60-170 万处理 token（推导估算，R2 实测校准）；LightRAG 钉死固定 tag（一周三发，已核验）
- 依据：LightRAG（MIT）PGGraphStorage 源码级确认基于 AGE 的 cypher()；检索阶段成本：MS GraphRAG global search 论文场景约 61 万 token/查询 vs LightRAG 关键词生成与检索 <100 token（**限定：LightRAG 作者方测算、仅检索阶段、不含最终送入 LLM 的上下文**，arXiv 2410.05779v3 原文核验）；MS GraphRAG 社区摘要架构反增量（官方源码证实），排除

**Story 4.4.2 wiki 页 = 图的物化投影（全网无先例）**
- Given 图中存在实体节点
- When 系统渲染该实体的 wiki 页
- Then wiki 页是图的**物化投影**：持久存储 + git 落盘，图为唯一权威源；图增量变更时受影响页重新渲染并更新物化副本（含 git commit）；页内 `[[链接]]` 由图的边生成——一条抽取管道同时喂图谱和 wiki，无第二套真相源
- 边界："接管编辑"段落锁定为人工事实（高置信标记）回写图，冲突时人工事实优先；回写以 diff 审计；物化语义保证页面历史/Obsidian 直开/接管编辑全部有载体
- 依据：双引擎融合全网无成熟项目，核心创新；R1 PoC 于第二步设计期完成

**Story 4.4.3 图谱可视化**
- Given 用户打开图谱视图
- When 渲染
- Then 全库"宇宙视图"用 sigma.js v3 WebGL + **服务端 NetworkX ≥3.4 内置 `forceatlas2_layout`（BSD）预布局坐标**；检索结果给 1-2 跳子图局部视图；LOD 缩放显隐、邻域高亮、按社区/度数/时间过滤渐进披露；孤页/枢纽页高亮（呼应 Lint）
- 边界：性能基线——1 万节点在 Chrome 最新版 + 集成显卡（M1/Iris Xe 级）下 ≥30fps；点击节点直达 wiki 页；可导出 JSON/交互式 HTML；**fa2/fa2_modified（GPLv3）与 igraph（GPL-2）禁入交付物**（许可红线，二轮核验）
- 依据：sigma.js 3.0.3 活跃（2026-06-09 仍 push）；NetworkX forceatlas2_layout 官方文档确认

### 4.6 记忆系统（四段式工程共识）

**Story 4.6.1 对话沉淀记忆**
- Given 用户与系统的全部对话
- When 后台异步记忆管线运行（不阻塞对话）
- Then 四段式：结构化事实抽取（带 schema）→ LLM 决策 ADD/UPDATE/DELETE/NOOP 冲突消解 → 时间衰减 + 周期整理任务（默认每周，整理产物进记忆面板可见）→ 记忆全部进用户可见面板
- 边界：检索"按需工具式"而非全量画像注入；轻量"近期摘要"预注入做个性化层；记忆条目带来源对话链接与置信度
- 依据：Mem0/Letta/LangMem 与 ChatGPT/Claude（**社区逆向分析非官方，可能随版本失效**）殊途同归的四段式；LoCoMo 数字全部不采信

**Story 4.6.2 记忆面板（用户主权）**
- Given 用户打开记忆面板
- When 浏览/编辑
- Then 按 semantic/episodic/procedural 三类陈列；每条可编辑/删除/置顶；删除 = 硬删除（含时序层对应节点/边）；记忆时间线可视化
- 边界：**记忆归属 per-user，永不跨用户共享**（共享库对话产生的记忆仍归发起人）
- 依据：可见可编辑已是用户预期底线，也是隐私卖点

**Story 4.6.3 时序记忆层（自研双时态，全 SKU 可开启）**
- Given 库设置开启时序记忆层
- When 记忆事实随时间演化
- Then **自研 Graphiti 式双时态建模，落本项目 AGE 图**：记忆边带事件时间/摄入时间双时间线属性 + episode 表保留原始对话指针；系统自动矛盾消解用"边失效"标记（t_invalid 置位）而非删除，演化历史可回溯
- 边界：用户主动删除记忆时对应节点/边硬删除（"边失效"仅用于系统自动消解）；CPU 本地档结构化输出质量不达标时如实降级标注并建议外接模型（同 R4 约束）
- 依据：**v1.0.2 必改项**——Graphiti 官方 driver 仅 Neo4j/FalkorDB/Neptune/Kuzu（源码枚举核验），无 AGE/Postgres 支持且 feature request 长期无人认领；FalkorDB 方案破坏单 PG 库原则且 SSPL 许可有分发风险；故砍掉 Graphiti 依赖、自研其双时态范式（范式本身来自 arXiv 2501.13956，已核验）——保住单库与许可干净两个卖点

**Story 4.6.4 记忆唤回**
- Given 新摄入内容与既有记忆/知识相关
- When 用户次日回访
- Then 主动提示"你之前提到的 X，新文档里有相关内容"
- 边界：相关度阈值 Schema 可调、每日提示上限默认 3 条、可一键关闭

- When 管理员在后台配置页接入
- Then 六路统一收口：OpenAI 兼容 / Anthropic / Gemini / 国产大模型 / 客户内网 vLLM/Ollama / 离线包内置引擎（llama.cpp）；页面化填参即生效

**Story 4.10.2 网关数据面（Go）+ 控制面（Python）**

**Story 4.10.3 全离线增强包（独立 SKU 规格，硬件账经二轮实测核验修订）**
- Given 涉密/断网客户
- When 部署离线包
- Then 标准镜像 + 模型包。**CPU 档**：模型合计约 6.5-7GB（Qwen3-4B Q4_K_M 2.5GB + Qwen3-Embedding-0.6B Q8 0.64GB + BGE-reranker-v2-m3 int8 ~0.6GB + SenseVoice int8 ~0.25GB + MinerU pipeline 1.5-2GB + MeloTTS-Chinese ~0.5GB + Docling 中文 OCR ~0.2GB，全部 HF API 实测体积）；**最低配置：8 物理核（需 AVX2，推荐 AVX-512/DDR5）/ 内存最低 24GB、推荐 32GB / 磁盘最低 64GB、推荐 128GB SSD**；llama.cpp 按"每模型一实例"编排（生成/嵌入/重排三个 server service）。**GPU 档**：15-29GB（加 4B/8B 档 + MinerU VLM + CosyVoice2 3.86GB + ColQwen2 可选），单卡 24GB 全承载。断网环境下库内全功能可用（摄入/检索/图谱/问答/记忆[时序层 CPU 档质量不达标时如实降级标注，见 R4]/Lint/MCP/音频概览-MeloTTS）
- 边界：天然依赖外网的功能（Web 搜索/云端连接器/在线短码激活）自动隐藏或置灰标注原因（离线 .forge 激活不受影响）；**CPU 档体验约束写死**：解析与 LLM 推理强制互斥排队、MinerU 用完即杀、PG shared_buffers ≤1GB、默认开 prompt cache（llama.cpp `--cache-reuse`）、RAG 注入上下文截断 ≤2K、默认关 thinking——预期体验如实公布：生成 8-14 t/s、长上下文首 token 20-60s（R14 性能表）；模型包独立分发（镜像外挂卷）
- 依据：二轮硬件账核验——**"8 核 16GB"被推翻**（MinerU 单项官方下限即 16GB、全家桶常驻 12-15.5GB）；24GB=互斥可用 / 32GB=与 MinerU 官方推荐对齐

## 6. 技术约束 & 决策（Tech Constraints & Decisions）

**关键技术决策**（以下编号条目以 §6.N 形式被全文引用；每条附依据，全部经两轮对抗性核验）：

1. **存储 = PostgreSQL 18 + Apache AGE 1.7.0 + pgvector 单库一路走到底（个人版与企业版同路线，用户拍板；v1.0.2 由 PG17 改定 PG18）**
   依据：**PG18 + AGE 1.7.0 是 LightRAG 官方基线**（其 `Dockerfile.postgres` = `pgvector/pgvector:pg18-trixie` + AGE `release/PG18/1.7.0` 源码编译，官方推荐镜像同此组合，原文核验）；AGE 官方 Docker `release_PG18_1.7.0` 双架构存在，而 PG17 的 1.7.0 镜像未发布、属无人验证区；pgvector 在 PG18 的性能回归（0.8.3 在修）仅涉 Hamming/Jaccard 二值距离，主线 halfvec+cosine 不受影响（CHANGELOG 核验），0.8.3 发布后升级。AGE 为 Apache 顶级项目、活跃（近 6 月 67 commit）；Azure 已 GA AGE+pgvector 同库组合。**版本锁死**：PG18 + AGE release/PG18/1.7.0 + pgvector 0.8.x + LightRAG 固定 tag；自建数据库镜像（4.13.1）。
   **AGE 能力边界**：openCypher 子集——发布版无 shortestPath/无 MERGE ON CREATE SET（源码 grammar 核验）；LightRAG upsert 不受影响；路径分析走 VLE+应用层，VLE 深度遍历纳入压测（R6）。"千万级向量单实例"无官方背书列为自测项。
   **项目级覆盖两项 Service 分类规则（记入 Terrane/.agent.md）**：① Database 锁 PG 引擎家族（postgres + openGauss/PolarDB-PG/KingbaseES 信创适配器，家族内多 provider 照做）；② Vector DB / Search 分类——向量与全文由 PG 内建承担（架构根基不可适配器化），外部 provider 不使用、`services/vector-db/` 不创建。services/ 实建清单：object-storage（全 provider）/ database（PG 家族）/ cache（redis+valkey）/ email（全 provider）。
2. **图引擎 = LightRAG（MIT），wiki 物化投影双引擎自研；时序记忆层自研双时态落 AGE（v1.0.2 砍掉 Graphiti 依赖——其官方无 AGE driver，源码枚举核验）**。已知坑（issues #1277/#2313/#2320）→ 锁 tag + 回归集。
